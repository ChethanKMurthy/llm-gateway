about the project

From a one-page spec
to a gateway that runs.

The brief was a "semantic cache." The interesting version is the whole control plane around it — so that's what got built: a complete, running, multi-provider LLM gateway, and the live console you can open right now.

The spec — a PRD/TRD — sketched fifteen features: dynamic routing, a multi-level cache, a prompt classifier, cost and latency prediction, guardrails, smart fallback, quality scoring, and a reinforcement-learning router. Most portfolio projects would implement the cache and call it a day. The premise here was different: build the entire thing, make it real, and make it inspectable.

The result is one nine-stage pipeline that every request flows through. A hit at the cache returns in about a millisecond. A miss gets classified, priced, routed to the model that's actually best for that kind of work, called with a health-aware fallback chain, scored, and learned from — and every one of those decisions is visible in the console's trace.

under the hood

The parts that matter.

A bandit, not a guess

The router is a Thompson-sampling contextual bandit. Intent is the context; candidate models are arms; reward blends quality, cost and latency. It needs no training set — it starts from informed priors and sharpens online toward whatever performs on real traffic.

Two layers of cache

L1 is an exact hash. L2 is embedding cosine over a single numpy matrix — one vectorized dot product per lookup, no vector DB needed. Thresholds are per-intent and self-tune: a low-quality hit makes that intent stricter.

Guardrails before the wire

Every prompt is scanned before it leaves the building: injection and jailbreak scoring, PII redaction, and a hard block on leaked secrets and API keys. A careless prompt can't exfiltrate a credential to a third party.

Observability, not vibes

Span-shaped events stream to the console over server-sent events: spend versus a frontier-only baseline, cache composition, latency percentiles, the learned policy, and provider health — all live.

real / simulated

What's real, stated plainly.

This audience can smell a faked demo, so here's the line. The algorithms are real and run on every request: the embeddings and semantic search, the classifier, the cache and its adaptive thresholds, the bandit, the circuit breaker, the cost math against a real 2026 price table, the guardrails.

The LLM responses are simulated by default — a faithful simulator with realistic latency, token counts, and failures — so the whole system runs offline with no keys. Add a provider key and that provider goes live; Groq calls, for instance, are real when a key is set. And the headline numbers come from intentionally cache-friendly demo traffic; real mixed traffic lands closer to a 20–45% hit-rate. The site says so on purpose.
